# 图式

## 图式（Schema）的详细描述

### 一、定义

**图式**是认知心理学中用于解释个体如何组织、存储和运用知识的核心概念。它指的是**一种具有抽象性、结构化、可重复使用的知识单元**，代表某一类事物、事件或情境的通用认知框架。简单说，图式就是人脑中关于"某类东西是什么样的"或"某类事情该怎么做的"心理模板。

例如，你有一个"图书馆图式"：知道图书馆里有书架、图书、管理员、安静的环境、借书需要办卡等。当你第一次走进任何一家图书馆时，这个图式会引导你快速理解环境并采取适当行为。

### 二、核心特征

1. **抽象性与概括性**  
   图式不是对单个具体经历的拷贝，而是从多个实例中提取出的共同特征。比如"狗"的图式综合了不同品种、大小、颜色的狗，而不仅仅是家里那只金毛。

2. **结构性与层级性**  
   图式内部包含固定的"槽位"（slots）和默认值。例如"餐厅图式"有槽位：菜品、价格、服务员、餐桌、菜单等。这些槽位可以填充具体信息（如"今天特价菜是披萨"）。图式之间可以嵌套——一个"生日派对图式"会调用"礼物图式"、"唱歌图式"、"蛋糕图式"。

3. **动态性与适应性**  
   图式不是僵化的。当遇到新信息时，可以通过**同化**（将新信息纳入现有图式）或**顺应**（修改图式以容纳新信息）进行调整。例如，儿童最初有"鸟会飞"的图式，见到企鹅后，可能调整图式，增加"有些鸟不会飞"这一分支。

4. **自动激活与引导作用**  
   一旦形成，图式会在相关情境中无意识地自动激活，指导注意、记忆、推理和行为。比如在教室里，你不假思索地去找座位坐下，因为"教室图式"告诉你要这么做。

5. **个体差异与文化差异**  
   图式依赖于个人经验和文化背景。一个从未见过雪的热带居民没有"雪地驾驶图式"；在有些文化中，"排队图式"并不存在。

### 三、主要功能

1. **信息选择与注意引导**  
   图式决定我们会注意什么。在拥挤的街道上，你的"过马路图式"会引导你注意红绿灯和车辆，而忽略广告牌上的文字。

2. **记忆组织与编码**  
   记忆不是录像，而是根据图式进行重构。我们更容易记住与已有图式一致的信息，也常常在回忆时用图式的默认值填补缺失细节（即"记忆的图式性重构"）。

3. **推理与预测**  
   利用图式填补信息空白，进行因果推断。例如，看到一个人进餐厅坐下、看菜单，你会推断他接下来会点菜——即使没有看到点菜过程。

4. **行为指导**  
   图式提供了行为脚本。你第一次参加学术会议，即使没经验，"会议图式"也能告诉你何时鼓掌、何时提问、如何和茶歇中的陌生人交谈。

### 四、常见类型（分类）

| 类型 | 描述 | 例子 |
|------|------|------|
| **客体图式** | 关于物体、人的特征 | "教授图式"：戴眼镜、说话慢、拿粉笔 |
| **事件图式**（脚本） | 关于常规事件的时间顺序 | "看病脚本"：挂号→候诊→描述症状→检查→取药 |
| **社会图式** | 关于社会群体、角色、规范 | "刻板印象"就是过度概括的社会图式 |
| **自我图式** | 关于自己个人特质的知识 | "我擅长数学"、"我害羞" |
| **策略图式** | 关于解决问题的方法步骤 | "遇到数学难题先画图" |

### 五、图式与心智模型的关系

在认知工程和人机交互领域，这两个术语常被交替使用，但存在细微区别：

- **图式**更偏向于**静态的知识结构**，是对世界"是什么"或"常规怎么做"的陈述性表征。例如"空调遥控器图式"：知道有模式键、温度加减键、风速键。
- **心智模型**则更强调**动态的、可运行的机制**，能够对系统进行**模拟、因果推理和预测**。例如，用户内心有一个"空调制冷原理"的心智模型（尽管可能不精确）：认为设定温度越低压缩机转得越快，所以会更费电。

**实际应用中**：图式是构成心智模型的"建筑材料"。多个相关的图式（客体图式、事件图式、因果图式）有机组合，就形成了用于理解特定人工系统的心智模型。

### 六、在认知工程中的应用

1. **界面设计**  
   设计者需要预测用户已有的图式。例如，电子书阅读器应借鉴"物理书图式"：左右翻页、书签标记位置。如果违背用户图式（如用上下滑动翻页），就会增加学习成本。

2. **可用性测试**  
   观察用户错误可以反推其错误图式。用户在一个文本框输入"密码"后按"Enter"，页面刷新但没有登录——可能用户有"表单提交图式"：认为在任何输入框按Enter就等于提交。发现后需要调整设计或提供提示。

3. **错误信息与指导**  
   当用户操作违背系统规则时，错误提示应帮助用户修正错误的图式，而不是仅仅报错。例如"文件不能保存在此处，请选择用户文件夹"比"访问被拒绝"更能帮助用户形成正确的存储图式。

4. **跨情境设计**  
   由于图式具有跨情境稳定性（如"保存"图式表现为Ctrl+S），设计应尽量保持一致，允许用户迁移已有的高效图式。

### 七、图式的获取与改变

- **形成**：通过反复暴露于同类情境，逐渐抽象出共性。儿童需要多次体验"推门进入"才能形成"推门图式"，区别"拉门图式"。
- **改变**：顽固的图式（尤其是刻板印象）改变困难，需要通过大量反例或反思性学习。在交互设计中，若要改变用户错误的图式，最好采用**渐进式引导**而不是强行纠正，例如通过分步教程和即时反馈。

### 八、总结

图式是认知的基本单元，它让人脑不必每次遇到新情境都从头学习，而是可以基于已有框架高效运作。在认知工程中，理解和运用图式是设计**易学、易用、符合用户预期**的系统的基础。设计者既需要尊重用户已有的图式（降低学习门槛），又需要在必要时巧妙地促使用户形成新的、更准确的图式。

## 心智模型的结构化表达

为了在认知工程（尤其是可计算系统）中明确描述与复用，心智模型可以拆解为以下六个核心维度。下面给出一个通用的结构化框架，并附上从 W23 数据（想法、意图、情境）中提炼的示例。

### 一、结构化框架（6要素）

| 维度 | 说明 |
|------|------|
| **1. 实体与属性** | 模型中包含哪些对象/概念？各有什么关键属性？ |
| **2. 因果关系** | 事件之间的因果链条（如果…那么…） |
| **3. 边界条件** | 模型适用的范围/前提/限制 |
| **4. 时间与动态** | 状态如何随时间变化（延迟、速率、顺序） |
| **5. 目标与行动映射** | 为了实现某个意图，会采取哪些操作？ |
| **6. 常见误解/简化** | 模型中不准确或过度简化的部分 |

### 1. 人机辩论收敛模型

**来源**：认知工程情境、2026-06-06 想法、think 意图

| 维度 | 描述 |
|------|------|
| **实体与属性** | - **人类**：具备长时间记忆和连续性，能跳出框架，负责评判和反馈<br>- **AI**：单次上下文短，自收敛会困在局部最优，负责大规模生成和辩论<br>- **辩论产物**：章程（contract），结构化的收敛结果<br>- **评分标准**：辩论的边界界定工具 |
| **因果关系** | - IF AI 自收敛到局部最优 THEN 人类反馈能使其跳出当前框架<br>- IF 辩论以产生章程为目标 THEN 辩论边界自然收敛<br>- IF 人类间歇性介入 THEN 收敛质量随轮次上升<br>- IF 辩论轮次足够多 THEN 产物从发散趋向结构化 |
| **边界条件** | - 需要至少两个 AI 角色进行辩论（可拓展监督）<br>- 人类介入的时机和频率需要校准（过早介入限制发散，过晚介入浪费 token）<br>- 适用于"需要章程/决议"的结构化问题，不适用于开放式创作 |
| **时间与动态** | - AI 自收敛 → 人类反馈（跳出框架）→ AI 在新框架下收敛 → 人类再反馈 → 直到人类也收敛为止<br>- 单个辩论轮次可在数分钟内完成<br>- 随轮次增加，收益递减 |
| **目标与行动映射** | - 意图"产生章程" → 启动人机辩论，设置评分标准作为边界<br>- 意图"跳出当前框架" → 注入人类反馈，提供外部视角<br>- 意图"验证收敛质量" → 递归评审评分标准 |
| **常见误解/简化** | - 以为"AI 辩论次数越多越精确"（实际上有收益递减拐点）<br>- 以为"人类只需在开始时设定目标"（实际上需要间歇性介入） |

### 2. 整合驱动上升循环模型

**来源**：叙事工程情境、2026-06-07 想法

| 维度 | 描述 |
|------|------|
| **实体与属性** | - **碎片**：未整合的发现、原型、概念（如母题、风格、故事）<br>- **整合动作**：丢弃过时概念、合并冗余结构、重组框架<br>- **清晰度**：整合后的认知状态，驱动下一轮发现<br>- **上升循环**：发现→整合→清晰→再发现的周期性过程 |
| **因果关系** | - IF 执行整合（丢弃+合并+重组） THEN 框架清晰度上升<br>- IF 框架清晰度上升 THEN 新发现更容易被识别<br>- IF 新发现被识别 THEN 产生新的需要整合的碎片<br>- IF 不执行整合 THEN 碎片堆积导致混乱，认知负荷上升 |
| **边界条件** | - 整合需要足够的碎片积累（过早整合无物可整，过晚整合成本过高）<br>- 适用于结构化知识领域，不适用于纯粹发散性创作<br>- 整合的"丢弃"操作需要明确的淘汰标准 |
| **时间与动态** | - 碎片积累阶段：多天到数周<br>- 整合阶段：数小时到一天（W23 叙事工程框架成型只用了一个傍晚）<br>- 整合后清晰期：可持续数天到一周<br>- 周期频率与实际产出速度正相关 |
| **目标与行动映射** | - 意图"提升框架清晰度" → 执行整合（列出所有碎片，逐项判断：保留/合并/丢弃）<br>- 意图"降低认知负荷" → 优先执行整合而非接受新碎片<br>- 意图"加速上升循环" → 主动设计整合节点，而非等混乱后被动整合 |
| **常见误解/简化** | - 以为"整合是收尾工作"（实际上是认知上升的核心驱动力）<br>- 以为"保留所有碎片更安全"（丢弃本身是上升的必要条件） |

### 3. 心理预算约束模型

**来源**：身心健康情境、多天想法（06-03/06-06/06-07）

| 维度 | 描述 |
|------|------|
| **实体与属性** | - **心理预算**：单个 POC 周期内允许的情绪能量投入上限<br>- **低耗能模式**：预算不足时的应对策略（AI 补全、降低决策标准）<br>- **无意义感**：预算耗尽的信号，高压探索的副产品<br>- **整合感**：预算恢复机制（"整合完以后脑子清楚多了"） |
| **因果关系** | - IF 心理预算充足 THEN 决策质量和产出效率正常<br>- IF 心理预算耗尽 THEN 出现无意义感、替代性抽离行为（如过度写作）<br>- IF 进入低耗能模式 THEN 可维持基本产出但无法产生高质量洞察<br>- IF 执行整合 THEN 心理预算部分恢复（清晰感是对冲机制） |
| **边界条件** | - 预算上限因人而异，即使同一人也因周而异<br>- 低耗能模式只能延缓耗尽无法阻止耗尽<br>- 心理预算是多个情境交汇的隐藏约束变量（影响研发、管理、健康三个看似不相关的领域） |
| **时间与动态** | - 从充沛到耗尽：约 3-4 天的高强度探索<br>- 恢复需要：至少半天的低强度活动或一次成功的整合<br>- 替代性抽离通常在第三天出现（W23：06-06 写一天小说） |
| **目标与行动映射** | - 意图"维持产出" → 在预算尚可时开 POC，在预算不足时切换到低耗能模式<br>- 意图"预防倦怠" → 主动设计恢复节点（整合日、低强度日）<br>- 意图"判断当前预算水平" → 观察是否有无意义感、是否出现抽离行为 |
| **常见误解/简化** | - 以为"疲劳管理就是早睡早起"（实际上无意义感比生理疲劳更早成为瓶颈）<br>- 以为"不同情境的心理预算独立"（三个不相干的情境共享同一个预算池） |

### 4. POC-收敛交替循环模型

**来源**：创新管理与认知工程情境的互补张力

| 维度 | 描述 |
|------|------|
| **实体与属性** | - **POC 范式**：大量小规模验证快速试错，做了再想（发散端）<br>- **意图工程**：结构化收敛，想清楚再做（收敛端）<br>- **循环**：发散→收敛→再发散→再收敛的交替过程 |
| **因果关系** | - IF POC 范式活跃 THEN 新想法和碎片快速涌现<br>- IF 意图工程介入 THEN 碎片被结构化为可执行的契约<br>- IF 两者交替运行 THEN 系统在探索和利用之间保持平衡<br>- IF 只有发散没有收敛 THEN 碎片堆积导致认知过载<br>- IF 只有收敛没有发散 THEN 创新停滞 |
| **边界条件** | - 需要两个方法论都已达到足够的成熟度才能形成有效循环<br>- 适用于需要"探索-利用"平衡的领域<br>- 循环周期的时长取决于领域复杂度 |
| **时间与动态** | - POC 阶段：数小时到数天（W23：默认日志 POC 当天见效）<br>- 收敛阶段：数天到一周（W23：意图工程概念一周内从发现到确立）<br>- 完整循环：1-2 周 |
| **目标与行动映射** | - 意图"快速验证假设" → 启动 POC（默认用日志做入口）<br>- 意图"结构化碎片" → 启动意图工程（提取意图→建模→契约化）<br>- 意图"维持循环健康" → 观察两端的产出比例，防止某一端过度主导 |
| **常见误解/简化** | - 以为"POC 和意图工程是两种可选方法"（实际上是一个循环的两个阶段）<br>- 以为"收敛应该在发散全部完成后才开始"（交替更有效） |

### 6. 抽象收益递减模型

**来源**：基础设施情境、2026-06-05 想法

|------|------|
| **实体与属性** | - **抽象**：从具体实例中提取通用模式（导航仪）<br>- **过度抽象**：为不存在的需求预支精力<br>- **系统成熟度**：决定当前应处于哪个抽象层级的度量 |
| **因果关系** | - IF 抽象水平与系统成熟度匹配 THEN 设计效率最优<br>- IF 抽象早于实际需求 THEN 维护负担增加且收益为零<br>- IF 抽象滞后于需求 THEN 重复劳动增加<br>- IF 遵循"先点后面，先具体后抽象" THEN 基础设施演进健康 |
| **边界条件** | - 适用于软件架构和系统设计领域<br>- 需要能够准确评估"当前系统成熟度"<br>- 抽象决策需要在有至少 3 个实例之后再做 |

| **常见误解/简化** | - 以为"抽象水平越高越好"（实际上抽象应当与实际成熟度匹配）<br>- 以为"提前抽象可以节省未来成本"（可能为不存在的需求预支精力） |

### 7. 元职能涌现模型

**来源**：元情境、身心健康情境（作为元意图）

| 维度 | 描述 |
|------|------|
| **实体与属性** | - **底层系统**：产生实际价值的业务领域<br>- **元职能**：标准化、心理健康等跨领域支持职能<br>- **成熟度**：底层系统的演化阶段，决定元职能是否需要独立 |
| **因果关系** | - IF 底层系统成熟度提升 THEN 元职能需求自然降低<br>- IF 元职能被人为独立 THEN 维护负担增加（需与底层系统保持同步）<br>- IF 元职能作为底层系统的天然产出 THEN 治理成本最低<br>- 标准化的终点不是规则文档，而是系统自我维护的元能力 |
| **边界条件** | - 适用于组织和系统治理场景<br>- 不宜主动推进元职能独立，宜随系统演化自然沉淀<br>- 元职能独立的标准是：底层系统已经稳定到可以支撑独立的治理层 |

### 三、与认知工程其他元素的接口

结构化表达的心智模型，可以直接用于：

- **预测用户行为**：给定情境+意图，利用规则推断用户会如何操作。
- **诊断错误**：当用户行为不符合系统逻辑时，比较其心智模型与规范模型（系统真实模型）的差异。
- **设计干预**：针对模型中的常见误解，设计教程、提示或强制功能。

**总结**：结构化心智模型 = **实体集 + 因果规则 + 边界条件 + 时间动态 + 行动映射 + 已知错误**。这种表达既可用于人工分析，也适合作为提示词的一部分输入给大模型进行模拟或推理。
